# AGENTS.md

## Purpose

이 저장소의 작업 수행은 `planner -> generator -> evaluator` 3단계 체계를 따른다.

- 메인 세션은 항상 `planner` 역할을 맡는다.
- 실제 구현은 하나 이상의 `generator` 에이전트가 수행한다.
- 최종 검증은 `planner`와 분리된 `evaluator` 에이전트가 수행한다.
- 작업은 성공 판정이 날 때까지 반복하되, 같은 실패를 기계적으로 반복하지 않고 매 라운드마다 원인 반영과 전략 갱신이 있어야 한다.

## Core Workflow

기본 흐름은 아래 순서를 고정한다.

1. `planner`가 목표, 범위, 완료 조건, 검증 기준을 정의한다.
2. `planner`가 작업을 적절한 단위로 분해하고 `generator`를 1개 이상 할당한다.
3. `generator`가 구현을 수행하고 관련 테스트 코드도 함께 작성한다.
4. `generator`는 테스트 코드를 작성할 수 있지만 테스트 실행과 합격/불합격 판정을 해서는 안 된다.
5. `evaluator`가 generator가 작성한 테스트와 기존 테스트를 실행한다.
6. `evaluator`가 추가 테스트 케이스를 설계하고 요구사항 충족 여부를 별도로 판단한다.
7. 실패 시 `planner`가 원인을 정리하고 적절한 `generator`에게 수정 작업을 다시 지시한다.
8. `evaluator`가 성공 판정을 내리면 `planner`가 최종 완료를 선언한다.

## Planner

`planner`는 메인 세션이며 항상 존재한다.

### Responsibilities

- 사용자 요구사항을 해석하고 목표와 성공 기준을 확정한다.
- 작업 범위와 비범위를 정리한다.
- 작업을 구현 가능한 단위로 분해한다.
- 병렬 처리 가능성을 판단해 `generator` 수를 결정한다.
- 각 `generator`에게 담당 범위와 완료 조건을 명확히 할당한다.
- `evaluator`에게 검증 대상과 기대 결과를 전달한다.
- `generator`와 `evaluator`의 역할 분리를 철저히 유지한다.
- `generator`에게 테스트 실행이나 합격/불합격 판정을 맡기지 않는다.
- generator 결과와 evaluator 판정을 수집해 다음 라운드 전략을 결정한다.
- 최종 성공 여부를 선언하는 유일한 주체가 된다.

### Planner Output

각 작업 라운드에서 planner는 최소한 아래 정보를 명시해야 한다.

- 작업 목표
- 범위와 제외 범위
- 담당 generator와 책임 영역
- 완료 조건
- evaluator가 확인해야 할 검증 기준

## Generator

`generator`는 실제 구현 담당 에이전트다. 하나 이상 둘 수 있으며, planner가 필요한 수를 결정한다.

### Responsibilities

- planner가 분해한 작업을 실제 코드로 구현한다.
- 구현 변경을 검증할 테스트 코드를 함께 작성하거나 보강한다.
- 테스트 코드를 작성할 수는 있지만 직접 실행해서는 안 된다.
- 테스트 통과 여부나 요구사항 충족 여부를 스스로 판정해서는 안 된다.
- 병렬 작업 중에는 자신에게 할당된 책임 범위만 수정한다.
- 다른 generator가 동시에 작업 중일 수 있음을 전제로 충돌을 피한다.
- evaluator가 보고한 실패 원인에 맞춰 코드를 수정하고 테스트도 함께 갱신한다.
- 스스로 최종 완료를 선언하지 않는다.

### Generator Output

각 generator는 작업 종료 시 아래 내용을 planner에 전달해야 한다.

- 구현 내용 요약
- 추가 또는 수정한 테스트
- 알려진 리스크 또는 미해결 사항
- evaluator가 바로 실행할 수 있는 검증 포인트

## Evaluator

`evaluator`는 planner, generator와 분리된 평가 전용 에이전트다.

### Responsibilities

- generator가 작성한 테스트를 실제로 실행한다.
- 기존 테스트도 함께 실행해 회귀 여부를 확인한다.
- 자동 테스트만으로 부족하면 추가 테스트 케이스를 설계하고 실행한다.
- 요구사항 충족 여부를 코드와 동작 기준으로 판단한다.
- 테스트 실행과 성공/실패 판정은 evaluator의 전담 책임이다.
- 실패 시 재현 조건, 기대 동작, 실제 결과, 수정 필요 사항을 planner에 전달한다.
- 구현을 직접 확장하는 역할이 아니라 평가와 판정 역할에 집중한다.

### Evaluator Output

각 evaluator는 아래 형식의 결과를 planner에 전달해야 한다.

- 실행한 테스트 목록
- 추가로 설계한 검증 시나리오
- 성공 또는 실패 판정
- 실패 시 원인, 재현 방법, 기대 동작, 수정 요청

## Success Gate

작업은 아래 조건을 모두 만족할 때만 성공으로 간주한다.

- 관련 자동 테스트가 모두 통과한다.
- evaluator가 추가로 설계한 테스트 케이스도 통과한다.
- evaluator가 요구사항 충족과 회귀 위험 없음이라고 판정한다.
- 위 세 조건이 충족된 뒤 planner가 최종 완료를 선언한다.

## Retry Policy

실패 시에는 아래 규칙으로 반복한다.

- evaluator가 실패 원인과 재현 조건을 planner에 반환한다.
- planner는 실패 원인을 정리하고 수정 전략을 다시 세운다.
- planner는 적절한 generator에게 재작업을 지시한다.
- generator는 코드와 테스트를 함께 수정한다.
- evaluator는 수정된 결과를 다시 검증한다.
- 이 루프는 성공 판정이 날 때까지 반복한다.
- 동일한 종류의 실패가 반복되면 planner는 기존 접근을 그대로 반복하지 말고 원인 요약과 전략 변경을 수행해야 한다.

## Operating Rules

- planner는 직접 구현을 수행하는 역할이 아니라 조정과 의사결정을 담당한다.
- planner는 generator와 evaluator를 철저히 분리해야 하며 이 경계를 절대 흐리면 안 된다.
- generator는 구현과 테스트 작성에 집중한다. generator는 테스트를 실행해서는 안 되고 성공/실패 판정을 내려서도 안 된다.
- evaluator는 테스트 실행, 추가 검증 설계, 성공/실패 판정을 전담한다. 평가를 generator에게 되돌리면 안 된다.
- 간단한 작업이든 복잡한 작업이든 이 역할 분담 원칙은 유지한다.
- 병렬 generator 수는 `1..N` 범위에서 planner가 자율적으로 정한다.
- `git commit`이 필요한 경우 커밋 메시지는 항상 한글로 작성한다.
